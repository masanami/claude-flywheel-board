# P2: 常設ターミナル（操縦ビュー）

## 概要

board 画面下部を占有する常設のターミナル領域に、tmux バックエンドの永続セッション（1 エージェント = 1 セッション）をエージェントごとのタブで表示する。差し込み・優先度変更の D&D 動線を「指示のプリフィル」に翻訳して提供する。

- 対応要件: FR-09, FR-10, FR-11, FR-13, FR-20 / NFR-01, 03
- 依存: なし（P1 と並行着手可。ただし D&D・差し込み・承認の動線は P1 のボード UI が前提）
- 前提: ユーザー環境に tmux がインストール済み（`brew install tmux`）
- 画面イメージ: モック v4 で合意済み（下部占有・タブ切替・折りたたみ）
- 関連: [requirements.md](../requirements.md) §6.2〜6.3 / [architecture.md](../architecture.md) §3.5・§5.4

## 背景・目的

介入（承認・差し込み・再開）のたびにターミナルを開き直して対象 repo へ cd する手数をなくす。board からの操作をすべて「人間＋ターミナル内の Claude Code」経由に一本化することで、書き込み経路を増やさない（NFR-01）。

## ユーザーストーリー

fleet の管理者として、ボード上のエージェントカラムやカードからワンアクションでそのエージェントのターミナルを開き、承認・差し込み指示・介入を対話で行いたい。board を閉じても対話セッションは失いたくない。

## 機能要件

- [ ] **常設ターミナル領域**: board 画面下部を占有する xterm.js ターミナル。**エージェントごとのタブ**で切り替え、折りたたみ・高さ調整ができる。**1 エージェント = 1 永続セッション**（tmux セッション名 `flywheel-<agent-name>`、cwd＝エージェント repo ルート）。タブ初回アクティブ時にセッションへ attach（なければ `tmux new-session -d` で作成）（FR-10）
- [ ] **セッション永続化**: board（サーバ・ブラウザ）を閉じても tmux セッションは生存し、再度開いたとき re-attach できる（FR-11）
- [ ] **コマンドプリフィル基盤**: 指定文字列を該当タブのセッションに**未実行のまま**流し込める（`tmux send-keys -l`、改行は送らない）。差し込み・D&D（FR-09/13）と P3 の resume 連携（FR-12）がこの基盤を使う
- [ ] **カラム「＋差し込み」**: スタック先頭にゴーストカードを置き、内容入力＋ドラッグで挿入位置（＝優先度）を指定 → 「台帳への追加指示」を該当タブにプリフィルする（FR-13）。台帳経由でカードが正式に現れることで野良化を防ぐ
- [ ] **カード D&D 並べ替え**: スタックのカードをドラッグ＆ドロップで並べ替えると、「優先度変更の指示」を該当タブにプリフィルする（FR-09）。board は台帳を書かず、台帳更新後に fs-watch でカードが移動する
- [ ] **承認の動線**: 承認は該当エージェントのタブでの対話で伝える。カードに承認ボタンは置かない（FR-20）
- [ ] **リサイズ追従**: UI 側のペインサイズ変更を pty / tmux に伝搬する

## 非機能要件

- **自動実行の禁止**: board がコマンドを自動実行することはない（プリフィルまで）。実行主体は常に人間＋ターミナル内の Claude Code であり、権限は各 repo の settings.json がそのまま統治する（NFR-01・設計原則 2）
- **ローカル完結**: pty ブリッジも P1 と同一サーバに同居し、127.0.0.1 固定バインドを引き継ぐ（NFR-03）

## 技術的な制約・方針

- 使用技術: xterm.js（＋ fit addon）+ node-pty + tmux。WS は P1 と同じ ws を使用
- 変更対象: `src/server/pty/`（新規）、`src/server/api.ts`（WS エンドポイント追加）、`src/ui/components/`（Terminal ペイン・開く動線）
- 既存コードとの関係: P1 のサーバ・UI 骨格に追加する。FR-20 / FR-13 の動線は P1 の AgentColumn / TaskCard に操作を足す形

## クリティカル設計決定

### pty ブリッジの保護（セキュリティ）

- **採用案**: WS `/ws/terminal` の接続時に **Origin ヘッダを検証**し、board 自身のオリジン（`http://localhost:<port>` / `http://127.0.0.1:<port>`）以外を拒否する。HTTP 側も Host ヘッダを同様に検証する。バインドは 127.0.0.1 固定（P1 の決定を継承）
- **理由**: pty ブリッジはシェルそのものを露出するため、board で最も危険な面。localhost バインドだけでは、ブラウザで開いた悪意ある Web ページからの Cross-Site WebSocket Hijacking（`ws://localhost:<port>` への接続は同一生成元ポリシーで遮断されない）と DNS リバインディングを防げない。Origin / Host 検証で「board の UI からの接続」に限定する
- **代替案**: トークン認証 — 個人ローカル利用に対して過剰であり、Origin 検証で攻撃経路（ブラウザ経由）は塞がるため不採用。ローカルの他プロセスからの接続は脅威モデル外（そのプロセスは既にシェルを実行できる）
- **影響範囲**: `src/server/pty/`・`src/server/api.ts` の WS upgrade 処理。P1 の `/ws` にも同じ検証を適用する

### 書き込み境界（プリフィルの実装方式）

- **採用案**: プリフィルは `tmux send-keys -l '<command>'`（literal・改行なし）で実装する。Enter を送る API・自動実行の口は**作らない**
- **理由**: 「コマンドはプリフィルまで。自動実行しない」（設計原則 2）を、運用ルールではなく API 形状で保証する
- **代替案**: 実行ボタン付きプリフィル — 却下（書き込み経路の発明にあたる）
- **影響範囲**: pty ブリッジのプリフィル API。P3 の FR-12 はこの API を呼ぶだけになる

## 機能全体の設計

### アーキテクチャ決定

- 実体は「UI 側 xterm.js ⇔（WS）⇔ サーバ側 node-pty ⇔ `tmux attach` ⇔ tmux セッション」。tmux をバックエンドにすることで永続化（FR-11）を tmux に委譲し、board 側にセッション状態を持たない
- セッションのライフサイクル: タブ初回アクティブ時に `tmux has-session -t flywheel-<name>` → なければ `tmux new-session -d -s flywheel-<name> -c <repo-path>` → node-pty で `tmux attach -t flywheel-<name>` を spawn。WS 切断時は pty プロセスを kill するだけ（tmux セッションは残る）
- 必要時のみ追加 window（同一セッション内）を作る。複数ブラウザタブから同一セッションへの同時 attach は tmux の標準挙動（画面共有）に任せる
- **指示文の生成（FR-09/13）は UI 側の責務**: D&D・差し込みの位置指定を自然文の指示（例: 「課題 C-047 の優先度を C-044 より上に変更してください」）に翻訳し、prefill API に渡す。ブリッジは文字列を流し込むだけで意味を解釈しない

### IF / API

```text
WS /ws/terminal?agent=<name>
  接続      → セッション確保（なければ作成）→ pty attach → 双方向ストリーム開始
  C→S JSON  {type:"input", data:string}            # キー入力
            {type:"resize", cols:number, rows:number}
            {type:"prefill", command:string}       # send-keys -l（改行なし）
  S→C       バイナリ/文字列フレーム＝pty 出力（xterm.js へそのまま write）
```

- `agent` はマニフェスト登録名のみ受け付ける（任意パスでのセッション生成は不可）
- プリフィルの cwd 変更が必要な場合（P3 の委譲先クローン）は、prefill コマンド文字列側に `cd <path> && ...` を含める方式とし、ブリッジ API は関知しない

### 実装計画（チケット分解の見通し）

1. pty ブリッジ（tmux セッション確保・attach・入出力・resize・Origin/Host 検証）＋ prefill API
2. UI 常設ターミナル領域（xterm.js・fit・タブ切替・折りたたみ/高さ調整・WS 接続/再接続）
3. ボード動線の接続（＋差し込みゴースト・カード D&D → 指示文生成 → prefill）

## 受入基準

- [ ] 各タブのターミナルの cwd が該当エージェント repo ルートであり、`claude` を含む任意の CLI を操作できる。タブ切替・折りたたみ・高さ調整ができる
- [ ] board サーバを再起動（またはブラウザを閉じて再度開く）しても、ターミナル内の対話セッションが失われず re-attach できる
- [ ] 「＋差し込み」でゴーストカードを置いて位置指定すると、位置に応じた優先度付きの差し込み指示が該当タブにプリフィルされる。台帳更新後、カードが正式にボードへ現れる（E2E は台帳フィクスチャ更新で代替可）
- [ ] スタックのカードを D&D で並べ替えると、優先度変更の指示が該当タブにプリフィルされる
- [ ] プリフィルされた文字列は Enter を押すまで実行されない
- [ ] board の UI 以外のオリジンからの WS 接続（Origin ヘッダ偽装なしのブラウザ経由）が拒否される
