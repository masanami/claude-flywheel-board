# 要件定義 — claude-flywheel-board

> claude-flywheel の fleet（複数の自律エージェント）を 1 画面で観測・操縦するローカル GUI に対する要件を定義する。
> アーキテクチャ・実装方式は別ドキュメント（[architecture.md](./architecture.md)）で扱う。本書は **何を満たすべきか（What）** に集中する。

## 1. 目的

### 最上位の目標（North Star）

**「だれが・どのタスクを・いまどうしているか」を 1 画面で即答できるようにすること。**

複数エージェントの同時稼働と差し込みタスクが常態の運用で、状況把握のためにファイルを開いて回るコストをゼロにする。あわせて、把握した状況に対する介入（承認・差し込み・停止セッションの再開）を同じ画面から行えるようにする。

### 提供物

- **(a) fleet ボード** — エージェントごとの縦カラムにタスクを積んだ観測ビュー。
- **(b) 埋め込みターミナル** — ボードから直接 Claude Code CLI を操作する操縦ビュー。

## 2. 背景・課題

- claude-flywheel の状態はファイル正本（課題台帳・journal・memory）で、エージェント repo ごとに分散している。**横断的な現況把握には全 repo のファイルを開いて回る必要がある**。
- 課題台帳のステータスはサイクル単位でしか更新されず、**委譲中の `claude -p` セッションが「いま」何をしているかはどこにも見えない**。
- **差し込みタスク**は台帳を経由しないと、どの管理面からも不可視になる（野良化）。
- 介入（承認・再開・差し込み）のたびにターミナルを開き直し、対象 repo へ cd する手数がかかる。

→ ファイル正本という設計は維持したまま、その上に**読み取り専用の投影（ボード）と操作の入口（ターミナル）**を被せる。

## 3. スコープ

### 3.1 対象（In Scope）

| 領域 | 具体例 |
| --- | --- |
| 観測 | fleet 横断のタスクボード、実行中セッションの一覧、承認待ちの集約表示 |
| 操縦 | 埋め込みターミナルからの Claude Code CLI 操作、タスクカードからのセッション再開、差し込みタスクの入口 |
| データ整合 | 台帳・イベントログのパースエラー可視化（正本の欠陥を隠さない） |

### 3.2 対象外（Out of Scope）

| 項目 | 理由 |
| --- | --- |
| GUI からの状態ファイル直接編集（台帳編集 UI・承認ボタン等） | 書き込み経路を増やさない（NFR-01）。承認は対話（ターミナル内の Claude Code セッション）経由が正規動線 |
| モバイル対応・リモートアクセス・チーム共有 | 利用者は手元のマシンの個人 1 名（§5）。localhost 完結 |
| 課題台帳の DB 化（正本の置き換え） | 正本はファイルのまま（claude-flywheel 側の設計）。DB はボード内部の読み取りキャッシュに限る |
| run-cycle 等の自走制御（スケジューラ・キュー管理） | 制御プレーンは claude-flywheel の責務。board は観測プレーン＋操作の入口に徹する |
| `runs.jsonl` 等フォーマット仕様の正本管理 | 仕様の正本は claude-flywheel 側 docs（§8） |

## 4. 用語定義

| 用語 | 定義 |
| --- | --- |
| fleet | claude-flywheel で作る、プロジェクトごとに独立した自律エージェントの群 |
| エージェント | fleet の 1 体。独立リポジトリに課題台帳・positions・memory・runtime を持ち自走する |
| 課題台帳 | 各エージェント repo の `challenge-ledger.md`。課題の正本（ステータス: 未分類→分類済→計画承認待ち→着手中→検証中→完了確認待ち→完了） |
| runs.jsonl | 各エージェント repo の `.flywheel/runs.jsonl`。サイクル・委譲セッションの start/end イベントログ（claude-flywheel 側 P0 で仕様化） |
| 制御プレーン | 自走を回す側（claude-flywheel: run-cycle・スキル・cron） |
| 観測プレーン | 状態を映す側（本アプリ）。制御プレーンに影響を与えない |
| 差し込みタスク | 定常サイクル外で人間が割り込ませるタスク。台帳 or runs.jsonl を経由しない限り不可視 |
| fleet マニフェスト | board に登録するエージェント repo（ローカルパス）の一覧 |

## 5. アクター

| アクター | 役割 |
| --- | --- |
| 管理者（人間・個人 1 名） | board の唯一の利用者。手元のマシンで起動し、観測・承認・差し込み・介入を行う |
| エージェント群 | 直接のアクターではない。ファイル（台帳・runs.jsonl・journal）を介して間接的に board へ状態を供給する |
| Claude Code CLI | 埋め込みターミナル内で人間が操作する。board からの操作はすべてこの CLI を経由する |

## 6. 機能要件（FR）

### 6.1 fleet ボード（観測）

- **FR-01 fleet 登録**: fleet マニフェストで複数エージェント repo（ローカルパス）を登録でき、board が全 repo をスキャンする。
- **FR-02 エージェント縦カラム**: カラム＝エージェントとし、各カラムは上から「実行中 → 承認待ち → スタック（優先度順）」の順で積む。カラムヘッダにはエージェント名とサイクル状態（実行中/idle/応答なし警告）を表示する。
- **FR-03 タスクカード**: `challenge-ledger.md` をパースし、課題 ID・タイトル・ステータス（バッジ表示）・優先度・担当ポジションをカードとして表示する。
- **FR-04 承認待ちの集約**: `計画承認待ち` / `完了確認待ち` の課題をカラム内で最上位グループにハイライトし、fleet 横断の「承認待ちのみ」フィルタを提供する。
- **FR-05 実行中セッション表示**: `runs.jsonl` から実行中（start があり end がないイベント）を検出し、課題 ID・委譲先 repo・経過時間を表示する。**一定時間を超えた実行中は「応答なし（要確認）」として警告表示**する（クラッシュした子セッションの検知）。
- **FR-06 ライブ反映**: 対象ファイルの変更（fs-watch）を検知し、ページ再読み込みなしでボードへ反映する。
- **FR-07 パースエラーの可視化**: 台帳・イベントログのパースに失敗したエントリは**黙って落とさず、エラーとしてボード上に表示**する（正本とボードの乖離が信頼を壊すため）。
- **FR-08 カード詳細・作業ログ**: カードのホバーで直近の作業要約を表示し、クリックで課題詳細（台帳の全項目）と**これまでの作業ログ**（journal / runs.jsonl を課題 ID で突き合わせた読み取り専用タイムライン）を表示する。
- **FR-09 位置＝優先度**: スタックは優先度順に表示する。カードのドラッグ＆ドロップによる並べ替え・差し込みの挿入位置指定は、**エージェントへの優先度変更・差し込み指示としてターミナルにプリフィル**される（実行は人間が Enter。台帳を書くのはターミナル内のエージェント＝NFR-01 維持）。

### 6.2 埋め込みターミナル（操縦）

- **FR-10 常設ターミナル**: board 画面下部を占有する常設のターミナル領域で、Claude Code CLI を含む任意の CLI 操作を行える。**1 エージェント = 1 永続セッション**（cwd＝エージェント repo）とし、**エージェントごとのタブ**で切り替える。領域は折りたたみ・高さ調整ができる。
- **FR-11 セッション永続化**: board（アプリ・ブラウザ）を閉じてもターミナル内のセッションは生存し、再度開いたときに re-attach できる（長時間の run-cycle と共存するため）。
- **FR-12 子セッション再開の連携**: **応答なし警告（FR-05）のカードとカード詳細に限り**、journal / runs.jsonl に記録された session_id を用いた再開コマンド（`claude -p --resume <session-id>`、cwd＝委譲先クローン）を該当エージェントのタブへプリフィルするボタンを置く。通常カードにはターミナル操作ボタンを置かない（カードは観測に徹する）。
- **FR-13 差し込みタスクの入口**: カラムの「＋差し込み」で内容を入力し、スタック上の挿入位置（＝優先度）を指定すると、台帳への追加指示がターミナルにプリフィルされる（FR-09 の適用）。差し込みが台帳 / runs.jsonl を経由してボードに現れることで、野良化を防ぐ。

### 6.3 承認の動線

- **FR-20 承認は対話経由**: 承認は該当エージェントのターミナルタブでの対話（Claude Code セッション）で伝える。承認待ちカードはハイライト表示（FR-04）で気づきを担い、承認ボタン等の操作は持たない。**board が承認状態を直接書き換える機能は持たない**（Out of Scope §3.2）。

## 7. 非機能要件（NFR）

- **NFR-01 正本非改変**: board は状態ファイル（台帳・journal・memory・runs.jsonl）に一切書き込まない。書き込みはすべて埋め込みターミナル内のセッションを通じて（＝既存の規律・権限設定のまま）行われる。
- **NFR-02 オプショナル性**: board の停止・障害が flywheel の自走（cron の run-cycle・委譲セッション）に影響しない。board は制御プレーンの依存にならない。
- **NFR-03 ローカル完結**: localhost でのみ提供する。認証は設けない代わりに外部バインド・外部送信をしない。利用者は手元のマシンの個人 1 名。
- **NFR-04 キャッシュ再構築可能**: board 内部の索引（DB/メモリ）は読み取りキャッシュであり、破棄しても正本（ファイル）から常に再構築できる。
- **NFR-05 フォーマット契約への追従**: 台帳・イベントログの解釈は claude-flywheel 側 docs の仕様を正とする。仕様にない独自解釈を board 側に持ち込まない。
- **NFR-06 起動の軽さ**: 追加インフラ（外部 DB・デーモン常駐）なしに、1 コマンドで起動できる。

## 8. 前提・依存（claude-flywheel 側）

board が成立するために、claude-flywheel 側に以下を依存する。仕様の正本はすべて claude-flywheel 側 docs に置く。

| 依存 | 状態 | 内容 |
| --- | --- | --- |
| 課題台帳フォーマット | 仕様化済み | `challenge-ledger-format.md`。課題 ID・ステータス・分類欄の形式 |
| `runs.jsonl` スキーマ | **未・P0 で仕様化** | サイクル・委譲の start/end イベント（ts / event / challenge / repo / session_id）。run-cycle への記録規律の追加を含む |
| journal 索引 | 仕様化済み | `journal/index.jsonl`（1 周 1 レコード） |
| 観測プレーンの位置づけ | **未・改訂待ち** | claude-flywheel architecture.md §7「専用アプリは作らない」を制御プレーン限定に改訂し、読み取り専用の観測アプリを許容する |

## 9. 段階導入

| フェーズ | 内容 | 依存 |
| --- | --- | --- |
| **P0**（claude-flywheel 側） | `runs.jsonl` スキーマ仕様化＋ run-cycle への記録規律追加 | なし |
| **P1** | fleet ボード: エージェント縦カラム＋台帳パース＋承認待ちハイライト＋カード詳細/作業ログ（FR-01〜04, 06, 07, 08 の journal 分, 09 の表示部分） | 台帳＋journal 索引（いずれも仕様化済み） |
| **P2** | 常設ターミナル: 永続セッション・タブ切替・差し込み/優先度変更の指示プリフィル（FR-09〜11, 13, 20） | なし（P1 と並行可。D&D 動線は P1 のボード UI 前提） |
| **P3** | 実行中パネル＋子セッション再開連携（FR-05, 12、FR-02 のサイクル状態、FR-08 への runs 統合） | P0 ＋ P2 |

## 10. 受け入れ基準（P1〜P3 完了時）

1. fleet の全エージェントについて「だれが・どのタスクを・いまどうしているか」がボードを開くだけで分かる（ファイルを開いて回らない）。カードのホバー/クリックで作業要約・作業ログまで辿れる。
2. 承認待ちの課題が見落とされず、ターミナルタブの対話で承認に入れる。
3. 応答なしの委譲セッションがボード上で検知でき、カードの再開コマンドプリフィルで介入できる。
4. 差し込みタスクがボード経由の動線（位置＝優先度の指定→指示プリフィル）で投入され、台帳経由でボードに現れる。
5. board を完全に停止しても run-cycle の自走が影響を受けない。
